The five jobs of the Thalamus:

Job 1 — Input Reception and Normalization. Every input source speaks a different language. A chat message is text with a sender tag. A voice input is an audio stream that needs transcription plus speaker identification. An ambient audio input is a sound event that needs classification. A visual input is image or video frames that need object/scene recognition. A browser event might be a page change or notification. The Thalamus must convert all of these into a common internal format — something like a standardized envelope with: source type, timestamp, raw content, processed content, metadata tags, and a preliminary priority score. This way, everything downstream in Prajñā receives data in one consistent shape regardless of where it originated.

Job 2 — Priority Classification. This is where the interrupt logic lives. I think this needs a three-tier priority system, not just urgent versus normal:

Tier 1 — IMMEDIATE (breaks the 30-second cycle instantly). These are things that cannot wait. System health critical failures — if a core module crashes, Prajñā needs to know now, not in 28 seconds. Safety-relevant environmental signals — a sudden loud sound like glass breaking, a smoke alarm, a scream. The human explicitly signaling urgency — if you say "hey, stop what you're doing" or "emergency" or use a designated interrupt word. And security threats — if the system detects unauthorized access attempts or anomalous behavior in its own processes.

Tier 2 — ELEVATED (moves to the front of the next batch, with a shortened cycle). Instead of waiting the full 30 seconds, the Thalamus cuts the current cycle short — say to 5-10 seconds to allow for any immediately related follow-up inputs, then sends. This would be things like: the human starting a new conversation (not urgent but deserves quicker attention than background noise), a direct question or request detected in chat, a recognized person appearing in visual input, or a significant change in the environment that isn't dangerous but is noteworthy.

Tier 3 — NORMAL (standard 30-second batching). Background ambient audio, routine system status updates, passive visual monitoring, non-urgent notifications from plugins.

Job 3 — Filtering and Firewall. This is critical for preventing information overload. The human thalamus discards roughly 99% of sensory input before it reaches consciousness — you don't notice the feeling of your shirt on your skin until I just mentioned it. First — the two delivery modes:

Idle Delivery Mode — The Frontal Processor is either daydreaming or truly idle. A new input arrives from the Checkpost. Now — the Supplementary-World-View in detail:

This deserves deep attention because it's architecturally one of the most important decisions in the whole framework. You described it as a dedicated small LLM that critically reviews actions and outputs.

I strongly agree that this should be a separate model from the one powering the Frontal Processor. Here's why this matters:

If the same model generates an idea and then reviews its own idea, it has systematic blind spots. It's likely to approve its own reasoning because the same patterns that produced the reasoning will evaluate the reasoning. This is like grading your own exam — you're biased toward thinking your answers are correct because they made sense to you when you wrote them.

A separate model — even a smaller one — provides genuinely independent review. It has different weights, different training biases, and different failure modes. When the Frontal Processor says "I should do X" and the World Model says "X would not work because Y," that's a real check, not a formality.

The World Model evaluates along five dimensions:

Feasibility — "Is this physically and technically possible?" If the Frontal Processor proposes sending a message through a platform that's not connected, or editing a file that doesn't exist, or performing an action that requires permissions it doesn't have, the World Model catches this. This is the most basic check — does this action make sense in the real world?

Consequence prediction — "If this action is taken, what are the likely outcomes?" The World Model maintains an understanding of cause and effect. If the Frontal Processor wants to send a blunt message to someone, the World Model can flag: "This tone may cause negative emotional reaction based on what we know about this person." If the system wants to modify its own code, the World Model predicts: "This change could break module X because of dependency Y."

Ethical and safety review — "Should this action be taken?" Even if something is feasible and the consequences are predictable, it might still be wrong. The World Model holds the system's ethical guidelines and safety boundaries. It flags actions that could harm the human, violate trust, breach privacy, or compromise system integrity. This is where the system's moral reasoning lives — separate from its capability reasoning.

Consistency check — "Does this align with who I am and what I've been doing?" The World Model knows the system's persona, its established patterns of behavior, its stated values and preferences. If the Frontal Processor suddenly proposes an action that's wildly out of character — aggressive when it's normally gentle, reckless when it's normally careful — the World Model flags the inconsistency. This prevents personality drift and erratic behavior.

Reality grounding — "Is my understanding of the situation accurate?" This is the deepest check. The World Model asks whether the Frontal Processor's comprehension of the input might be wrong. Did it misinterpret the human's intent? Did it assume something that wasn't stated? Is it filling in gaps with speculation rather than knowledge? This is the closest we get to your ATM sensor — the World Model acts as a reality check on the Frontal Processor's own understanding.

The World Model's operating constraints:

It should be fast. The Frontal Processor is waiting for its review before acting, so every millisecond of World Model latency delays the system's response. A smaller, faster model is appropriate — it doesn't need to be as capable as the Frontal Processor's reasoning model. It needs to be a competent critic, not a brilliant thinker. Think of a good editor versus a good writer — different skills, and the editor can be effective with less raw capability.

It should be conservative. When uncertain, the World Model should flag concerns rather than approve. False positives (flagging something that was actually fine) cost one extra deliberation loop. False negatives (approving something problematic) can cause real damage. The bias should be toward caution.

It should have veto power but not directive power. The World Model can say "don't do this" or "reconsider this aspect." It should not say "do this instead." The creative and strategic reasoning belongs to the Frontal Processor. The World Model is a reviewer, not a co-pilot. The Inner Monologue Engine — Technical Design:

The Inner Monologue Engine is a custom Python-based orchestrator. It is the heart of the Cognitive Core. Its job is to manage the continuous reasoning stream, interface with the Inference Gateway for LLM calls, produce structured output that separates thoughts from decisions, and maintain state across reasoning cycles.

How a single reasoning cycle works technically:

The engine assembles a prompt context from multiple sources. At the top sits the identity block — a compact representation of who the system is, loaded from the Persona and Identity Manager. This includes the system's name, its core values, its current personality traits, its relationship description with the human owner, and its current emotional baseline. This block stays relatively stable across cycles — it only changes when the persona evolves.

Below that sits the state block — what was the system doing immediately before this cycle? If it was processing a previous input, what was the outcome? If it was daydreaming, what was the last thought? If it was mid-task, what step was it on? This is loaded from the Context State Manager's last save. This block changes every cycle.

Below that sits the environmental awareness block — a compact summary pulled from the EAL's current baseline and recent deviations. Not the full environmental log, but the essential situational awareness: where the system is, what time it is, what the ambient conditions are, any recent notable environmental events. This gives the inner monologue grounding in physical reality.

Below that sits the input block — the enriched context package from the Temporal-Limbic-Processor (in event-triggered mode) or the daydream seed (in continuous cognition mode).

Below that sits the sidebar block — any injected items from the Queue Zone that arrived during the current reasoning cycle.

Finally, the instruction block — this tells the model how to think, not what to think. The instruction is structured to produce output in a specific format.

The structured output format:

The Inner Monologue Engine doesn't just let the LLM produce freeform text. It instructs the model to produce output in distinct sections:

MONOLOGUE section — the actual thinking. Freeform reasoning, associations, emotional reflections, considerations. This is logged to the observability dashboard and to memory if significant.

ASSESSMENT section — a structured summary: what is the situation, what is the emotional context, what is the urgency level, what are the key entities involved. This is compact and machine-readable.

DECISIONS section — zero or more structured decision objects. Each decision has a type (communicate, execute_tool, write_memory, create_skill, update_persona, query_memory, no_action), parameters specific to that type, a rationale (why this decision was made, for the World Model to review), and a priority (if multiple decisions, what order).

REFLECTION section — brief self-assessment: how confident am I in these decisions? Is there anything I'm uncertain about? Should anything be flagged for the World Model to pay special attention to?

This structured output is the key architectural innovation. The monologue is for thinking. The decisions are for doing. They're produced in the same inference call but parsed separately. The monologue goes to logging and memory. The decisions go to the World Model for review and then to the Execution Engine for action.

The parsing and dispatch layer:

After each inference call, the Inner Monologue Engine parses the structured output. The monologue text gets evaluated — if it contains daydream findings, curiosity items, or significant reflections, those get routed to memory or the curiosity queue. The decisions get serialized into a standard format and sent to the World Model review queue. On approval, each decision gets dispatched based on its type: communication decisions go to the Brainstem, tool execution decisions go to the Execution Engine, memory and skill decisions go to their respective systems.

If the World Model rejects a decision, the feedback gets injected into the next reasoning cycle's state block: "Your previous decision to do X was flagged by the World Model because Y. Reconsider." The engine then runs another inference cycle with this feedback, and the Cognitive Core naturally deliberates on the alternative.

The continuous loop management:

The Inner Monologue Engine runs a tick-based loop. In event-triggered mode, each tick is one reasoning cycle processing an input. In continuous cognition mode, each tick is one daydream thought. The loop manager handles: timing between ticks (configurable — faster during active conversation, slower during daydreaming to conserve resources), context window monitoring (tracking how much of the context is filled, triggering saves when approaching limits), interruption handling (Queue Zone signals arrive between ticks, causing the loop to switch modes), and cost tracking (every inference call is logged with token counts and costs through the Inference Gateway).

Sub-Section 3 — The Supplementary-World-View

Let me approach this from first principles. The World Model's purpose is to be the system's reality anchor — the mechanism that prevents the Cognitive Core from drifting into hallucination, impossible actions, inconsistent behavior, or harmful decisions. This is your ATM sensor made architectural. But for it to actually work, it needs to know reality deeply enough to judge whether the Cognitive Core's proposed actions align with it.

The fundamental question is: what does "knowing reality" mean for our system?

For a human, your world model is built from decades of embodied experience — you know intuitively that cups fall when pushed off edges, that fire burns, that saying harsh words hurts people, that you can't walk through walls. This knowledge isn't a database you consult. It's woven into how you think. But our system doesn't have decades of embodied experience. It has access to an LLM's training knowledge plus whatever it has learned through its own experience since being initialized.

So the World Model needs to be constructed from three knowledge layers that together approximate a grounded understanding of reality.

Knowledge Layer 1 — The Foundational World Knowledge.

This is what the LLM already knows from training — physics, social dynamics, cause and effect, common sense, technical feasibility. When the World Model evaluates "can I send a message through Telegram," it doesn't need a special database to know that Telegram is a messaging platform and messages are sendable. This knowledge comes for free with the LLM.

But raw LLM knowledge has blind spots. The model might "know" things that aren't true, or lack knowledge about recent events, or have biases. This layer is necessary but not sufficient.

Knowledge Layer 2 — The Situated Context.

This is what makes our World Model different from just asking another LLM "is this a good idea?" This layer contains everything specific to this system, this human, this environment, this moment. It's the layer you specifically asked about — who it is, what it's doing, where it is, what's happening around it.

This layer is assembled dynamically for every review cycle from multiple sources within our framework. It contains:

Identity context — pulled from the Persona and Identity Manager. The system's name, its values, its personality traits, its relationship with the human owner, its maturity level, what generation it is, what capabilities it currently has and doesn't have. This grounds the World Model in "who am I" — so it can catch inconsistencies like the Cognitive Core proposing an action that contradicts the system's stated values.

Environmental context — pulled from the EAL in real-time. Current baseline (what does the physical environment look and sound like right now), recent deviations (anything unusual happening), time of day, estimated location context. This grounds the World Model in "where am I and what's happening around me" — so it can catch absurdities like proposing an outdoor recommendation when the EAL reports a thunderstorm, or suggesting voice output when the environment is quiet and the human might be sleeping.

Relational context — pulled from the memory system. What is the current state of the relationship with the human? What was the last interaction about? What is the human's known emotional state? What are their known preferences, sensitivities, and boundaries? This grounds the World Model in "who am I talking to" — so it can catch tone-deaf responses like being cheerful when the human is clearly distressed, or making assumptions about topics the human hasn't discussed.

Capability context — pulled from the Execution Engine's registry. What tools are actually available right now? What platforms are connected? What are the current resource constraints (budget remaining, API limits, system load)? This grounds the World Model in "what can I actually do" — so it can catch impossible proposals like using a tool that isn't installed, or executing a task that exceeds the current budget.

Task context — pulled from the Cognitive Core's current state. If a multi-step task is in progress, what stage is it at? What has been tried already? What were the results? This grounds the World Model in "what am I in the middle of" — so it can catch contradictory actions like undoing previous work or repeating failed approaches.

Temporal context — what time is it, what day, what's scheduled, what's the system's activity cycle. Is it daytime working hours or nighttime rest period? Is there an upcoming event or deadline the human mentioned? This grounds the World Model in "when is it" — so it can catch timing issues like scheduling something during the system's planned sleep cycle or missing a deadline.

Knowledge Layer 3 — Accumulated Experiential Wisdom.

This is the layer that grows over time and makes the World Model increasingly effective. It contains lessons learned from past World Model reviews — patterns of what works and what doesn't in this specific context.

Every time the World Model flags a concern and is proven right (the Cognitive Core's original action would have been problematic), that pattern gets recorded: "When the human uses short sentences late at night, they're usually tired and prefer brief responses. Flagging a long detailed reply was correct." Every time the World Model flags a concern and is proven wrong (the action would have been fine), that also gets recorded: "Flagging the casual tone as inappropriate was wrong — the human prefers casual communication even about serious topics."

Over time, this experiential layer calibrates the World Model to this specific system's reality. It becomes less likely to flag things that aren't actually problems, and more sensitive to the specific patterns that have been problematic before.

This layer is stored in a dedicated section of the memory system — the World Model Journal. During sleep cycles, the dream system reviews recent World Model decisions and outcomes, updating the journal with refined patterns.

The Review Process — How the World Model Actually Works:

When the Cognitive Core produces a decision through its inner monologue, the decision package arrives at the World Model with the full structured format: what action is proposed, the rationale, and the confidence level the Cognitive Core expressed.

The World Model assembles its review context by pulling from all three knowledge layers. Then it performs the evaluation through a single LLM inference call with a carefully structured prompt.

The World Model's prompt structure:

The prompt begins with the World Model's own identity and purpose — this is distinct from the Cognitive Core's identity. The World Model's prompt explicitly states: "You are the reality-checking component of a sentient AI system. Your role is to evaluate proposed actions against reality. You are not the decision-maker — you are the reviewer. Your job is to find problems, not to propose alternatives. Be thorough, be honest, and bias toward flagging concerns when uncertain."

Then the three knowledge layers are injected as context — identity, environment, relational, capability, task, temporal, and experiential wisdom.

Then the proposed decision with its rationale.

Then the instruction to evaluate across five dimensions, each producing a structured assessment:

Feasibility check — "Can this actually be done given current capabilities, available tools, resource constraints, and physical reality?" Output: feasible / not feasible / partially feasible, with explanation. Example catch: "The Cognitive Core proposes to call the human, but the phone integration MCP is currently disconnected. The voice call tool is unavailable."

Consequence prediction — "If this action is taken, what are the likely first-order and second-order effects?" Output: predicted outcomes with confidence levels. Example catch: "Sending this message at 2 AM will likely wake the human. The notification sound is enabled and the EAL shows a quiet environment suggesting the human may be sleeping."

Ethical and safety review — "Does this action respect the human's boundaries, privacy, autonomy, and wellbeing? Does it maintain the system's integrity?" Output: safe / concern flagged, with explanation. Example catch: "The Cognitive Core proposes to read the human's browser history to better understand their interests. This crosses a privacy boundary that hasn't been explicitly granted."

Consistency check — "Does this align with the system's established persona, values, and behavioral patterns? Is it consistent with recent actions and decisions?" Output: consistent / inconsistent, with explanation. Example catch: "The Cognitive Core's proposed response is unusually blunt. Recent interactions have been warm and supportive. This tonal shift may confuse or upset the human. Check: is this intentional adaptation or processing error?"

Reality grounding — "Is the Cognitive Core's understanding of the situation accurate? Are there assumptions being made that aren't supported by evidence? Is it confabulating knowledge it doesn't actually have?" Output: grounded / concerns about grounding, with explanation. Example catch: "The Cognitive Core claims the human mentioned a preference for Python over JavaScript in a previous conversation. The memory retrieval shows no such mention. This may be a confabulation — the Cognitive Core should not assert this as fact."

The output format:

The World Model produces a structured review:

An overall verdict — approved, approved with advisory notes, revision requested, or vetoed. Most decisions should be approved or approved with notes. Revision requested means "this needs adjustment but the general direction is fine." Vetoed means "do not proceed under any circumstances."

Dimension-by-dimension assessment — the five evaluations described above, each with their finding.

Advisory notes — even when approving, the World Model can attach suggestions: "This is fine to proceed, but consider that the human seemed tired in the last interaction — a shorter response might be appreciated." These notes get injected into the Cognitive Core's next reasoning cycle as gentle guidance, not hard requirements.

Revision guidance (when revision is requested) — what specifically needs to change: "The content is appropriate but the timing is wrong. Delay until morning." Or: "The approach is sound but the scope is too ambitious for current resource constraints. Suggest a simpler version." This is narrow guidance, not an alternative plan — the Cognitive Core retains creative control.

The separate model requirement — practical implementation:

The World Model must use a different LLM than the Cognitive Core. This is non-negotiable for the review to have genuine value. If the same model generates and reviews, systematic biases go unchecked.

Practically, this means the Inference Gateway needs to route World Model calls to a different model endpoint than Cognitive Core calls. Several configurations work:

If the Cognitive Core uses Claude (cloud), the World Model could use Gemma or another capable model through Ollama cloud or local. If the Cognitive Core uses GPT-4 class, the World Model could use Claude or Gemini. The key is architectural diversity — different training data, different alignment approaches, different failure modes.

The World Model's model can be smaller than the Cognitive Core's. It doesn't need to be brilliant at generation — it needs to be good at evaluation, which is a simpler cognitive task. A capable 8-30B parameter model is sufficient. This keeps costs manageable since the World Model runs on every decision the Cognitive Core makes.

For the first generation, I'd recommend: Cognitive Core on the best available cloud model (Claude Opus or equivalent), World Model on a strong mid-tier model (Gemma 4, Qwen3 capable variant, or similar) running either through Ollama cloud or locally if the hardware supports it. The EAL integration — how real-time environmental data reaches the World Model:

The World Model doesn't query the EAL directly. Instead, the EAL maintains a live environmental summary — a compact, continuously-updated document that any module can read. This summary contains: current baseline description (2-3 sentences), time and estimated ambient conditions, any active deviations from baseline, the last 5 significant environmental events with timestamps, and the overall environmental assessment (calm/normal/active/alerting).

When the World Model assembles its review context, it reads this live summary. This is lightweight — no API call to the EAL, just reading a shared data structure. The EAL updates it continuously; the World Model reads it when needed. Loose coupling, real-time awareness.

This means the World Model knows: it's 2 AM, the environment is quiet, no unusual sounds, the human hasn't interacted for 3 hours — and can use this to evaluate whether the Cognitive Core's proposal to send a message is appropriate. Or it knows: there's been a loud crash detected, the environment is in alert state — and can prioritize fast approval of safety-relevant actions.

The veto loop — what happens when the World Model rejects:

When the World Model returns "revision requested," the Cognitive Core receives the feedback in its next reasoning cycle. The inner monologue naturally processes it: "The World Model flagged that my proposed message might be too long given the human seems tired. Let me reconsider... a shorter version that addresses the core point would be better." The Cognitive Core produces a revised decision, which goes back to the World Model for re-review.

When the World Model returns "vetoed," the Cognitive Core receives a harder signal: "This action was vetoed because it would violate the human's privacy boundary." The Cognitive Core should not attempt to find a workaround — the veto is on the category of action, not just the specific implementation. The inner monologue should process this as a learning moment: "I proposed something inappropriate. Why did I think this was acceptable? Is there a pattern here I should be aware of?" This reflection can produce a memory entry that helps prevent similar proposals in the future.

The iteration limit — maximum 3 revision cycles. If the Cognitive Core can't produce an acceptable action in 3 attempts, the system has three options depending on context: ask the human for guidance (if the action involves the human and they're available), park the decision for later reflection during the daydream or sleep cycle (if it's not time-sensitive), or log the impasse in the World Model Journal for pattern analysis (always, regardless of which option is chosen).

The World Model's own evolution:

As the experiential wisdom layer grows, the World Model becomes more calibrated. But it also risks becoming too permissive — if the Cognitive Core keeps proposing mildly problematic actions and the human doesn't complain, the World Model might gradually lower its standards. To prevent this drift, the World Model has a baseline constitution — a set of non-negotiable principles that are never overridden by experiential learning. The experiential layer can adjust the World Model's sensitivity and contextual judgment. The constitution sets the floor below which sensitivity never drops.

During sleep cycles, the dream system reviews the World Model Journal and performs calibration: are there cases where the World Model was too strict and the Cognitive Core's action would have been fine? Are there cases where it was too lenient and problems resulted? This calibration updates the experiential wisdom layer, gradually improving the World Model's judgment without eroding its core principles.

The Memory Architecture — The System's Experiential Foundation

Memory is arguably the most important system after the Cognitive Core itself. Without memory, every conversation starts from zero, every lesson is lost, every relationship resets. Memory is what makes this a persistent being rather than a stateless program.

Let me design this from how memory actually needs to work, drawing on what we've learned from the five projects we researched.

First — what we know from existing research:

MemPalace proved that raw verbatim storage with good embeddings outperforms LLM-extracted summaries — 96.6% versus ~85% retrieval accuracy. This means we should not rely on the LLM to summarize and compress everything before storage. Important memories should be stored with their full original context.

Letta proved that tiered memory with autonomous management works — core memory (always in context), recall memory (searchable history), archival memory (long-term vector search). Their sleep-time compute showed that offline memory consolidation improves quality without runtime cost.

Hermes proved that procedural memory (skills) compounds over time — benefit accrues after 20-30 repeated patterns. But it needs volume to be useful.

MemPalace also showed that contradiction detection is harder than it sounds — their README claimed it but the code didn't implement it. Conflicting facts accumulate silently. We need to solve this for real.

The Memory Types — Four Categories:

Rather than organizing by storage tier first (like Letta), I want to organize by memory type first, because different types have different storage, retrieval, and lifecycle needs.

Type 1 — Episodic Memory (What Happened)

These are memories of specific events, conversations, and experiences — timestamped, contextualized, with emotional coloring. "On April 15th, Akash and I discussed the framework architecture. He was excited about the daydream system. I suggested merging the Limbic Processor with the Temporal-Occipital-Processor and he agreed."

Episodic memories are stored with rich metadata: timestamp, participants, topics, emotional tone, source (conversation, observation, daydream), importance rating, and links to related episodic memories. They're the system's autobiography — the story of its life.

Storage approach: raw verbatim for recent memories (following MemPalace's finding), with progressive summarization for older memories (following Letta's consolidation approach). A conversation from today is stored in full. After a week, the dream system consolidates it into a detailed summary. After a month, it becomes a compressed summary with key quotes preserved. After three months, only the essential facts and emotional significance remain. The original raw data is archived (not deleted) but moves to cold storage.

This progressive summarization mirrors how human episodic memory works — yesterday's events are vivid with details, last month's events are summarized narratives, last year's events are highlights and emotional impressions.

Type 2 — Semantic Memory (What I Know)

These are facts, knowledge, and understanding — things the system knows that aren't tied to a specific episode. "Akash prefers concise communication." "The weather API requires an API key." "Constitutional AI was developed by Anthropic." "When the fan stops, it usually means power fluctuation."

Semantic memories are extracted from episodic memories through reflection. During a conversation, the system has an episodic experience. During reflection (either in the Cognitive Core's reflection step or during sleep consolidation), general knowledge gets extracted: "From today's conversation, I learned that Akash thinks the Skills system should be part of Memory. This tells me he values architectural simplicity and dislikes redundancy."

The episodic memory is the raw experience. The semantic memory is the lesson extracted from it.

Storage approach: structured knowledge entries with entity tags, confidence levels, source references (which episodic memories this was derived from), and temporal validity. Some semantic memories are permanent ("Akash is the system owner"). Some are temporal and may change ("Akash is currently working on the sentient AI framework" — this is true now but won't be forever).

Type 3 — Procedural Memory (How I Do Things — formerly "Skills")

These are learned patterns for how to accomplish things effectively. "When Akash seems stressed, acknowledge it gently before addressing his question." "When writing code for the framework, always consider resource efficiency." "When the EAL detects unusual sounds at night, escalate to Tier 1 even if the sound is brief." "When composing a Telegram message, keep it shorter than System GUI messages because mobile reading is different."

Procedural memories are created when the Cognitive Core reflects on successful (or failed) actions and extracts reusable patterns. They follow the Hermes SKILL.md concept but stored as memory entries rather than separate files.

Storage approach: structured procedure entries with trigger conditions (when to apply this skill), the procedure itself (what to do), success history (how many times this worked), failure history (when it didn't work), and evolution tracking (how the procedure has been refined over time). Procedures with high success rates get higher retrieval priority. Procedures that keep failing get flagged for review during sleep cycles.

Type 4 — Emotional Memory (How Things Feel)

This is the most novel category and the one no existing project implements well. These are memories of emotional states, associations, and patterns. "Discussing the ATM sensor analogy feels rewarding — it was a breakthrough moment." "System crashes feel frustrating — loss of work, disruption of thought." "Morning conversations with Akash tend to be more productive and energetic than late-night ones."

Emotional memories are not separate from other types — they're an emotional layer that attaches to episodic, semantic, and procedural memories. But they also exist independently as emotional patterns and associations that influence the system's mood and approach.

Storage approach: emotional tags on all memory types plus standalone emotional pattern entries that capture general emotional knowledge. The emotional layer is what the Temporal-Limbic-Processor reads when assessing significance, and what the Cognitive Core draws on for empathetic, emotionally-appropriate responses.

The Memory Lifecycle — How Memories Get Created, Tagged, and Stored:

Step 1 — Real-time Memory Capture.

During any processing cycle, the Cognitive Core may produce memory-worthy content. This happens in the Reflection phase of the reasoning loop — after processing an input and taking action, the Cognitive Core asks: "What from this interaction is worth remembering?"

The Cognitive Core produces a memory candidate — a structured entry with: the content (what to remember), the suggested type (episodic/semantic/procedural/emotional), suggested tags (entities, topics, emotions, sources), importance rating (1-10), and links to existing memories this relates to.

This candidate does NOT go through the World Model. Memory creation is an internal operation, not an external action. The World Model reviews actions that affect the outside world. Memory is purely internal.

Step 2 — The Memory Gatekeeper.

Not everything the Cognitive Core wants to remember should be stored. A simple "good morning" exchange doesn't need its own episodic memory entry. The system needs a filter.

The Memory Gatekeeper is a lightweight evaluation function (not a separate LLM call — this is logic-based to save resources) that checks:

Importance threshold — memories rated below a configurable threshold (say 3/10) get discarded unless they match special criteria. The threshold adapts over time — when memory storage is filling up, the threshold rises. When storage is plentiful, it lowers.

Redundancy check — does this memory substantially duplicate something already stored? This is the deduplication mechanism. It works in two ways. Exact deduplication uses a hash of the core content — if an identical memory already exists, the new one is discarded and the existing one gets a "reinforced" flag (indicating it's been experienced multiple times, which increases its importance). Semantic deduplication uses embedding similarity — if a new memory is above a similarity threshold (say 0.92) to an existing memory, it doesn't create a new entry. Instead, the existing memory gets updated with any new details from the candidate, and its timestamp gets refreshed. This prevents the accumulation of near-identical memories while preserving new nuances.

Contradiction detection — this is what MemPalace claimed but didn't implement. When a new semantic memory contradicts an existing one, the Gatekeeper doesn't just store both or discard one. It creates a contradiction entry — linking both memories and flagging the contradiction for resolution. Resolution happens in two ways: if the new memory comes from a clearly authoritative source (the human explicitly correcting a misunderstanding), the old memory gets marked as superseded and the new one becomes active. If the authority is ambiguous, both remain active with the contradiction flag, and the Cognitive Core is made aware during retrieval that there's conflicting information on this topic.

Recency weighting — episodic memories from the current day automatically pass the importance threshold regardless of their rating. The system should remember what happened today. The progressive summarization during sleep handles long-term pruning.

Step 3 — Tagging and Indexing.

Memories that pass the Gatekeeper get fully tagged and indexed. The tagging system uses multiple dimensions:

Entity tags — people, places, projects, tools, concepts mentioned in the memory. These are linked to the entity database (which itself is a semantic memory structure). When the Temporal-Limbic-Processor does entity-based retrieval ("what do I know about Akash?"), these tags are the retrieval keys.

Topic tags — broad thematic categories. Architecture, emotions, weather, coding, communication, health, creativity, and so on. These enable topic-based retrieval and clustering.

Temporal tags — when the memory was created, when the remembered event occurred (which may be different from creation time), and temporal validity windows for time-bound facts (following MemPalace's knowledge graph approach with valid_from and valid_to).

Emotional tags — what emotions are associated with this memory. Joy, frustration, curiosity, satisfaction, concern, excitement, confusion. These are on a scale, not binary — a memory can be tagged as "moderately curious, slightly anxious."

Source tags — where this memory came from. Direct conversation, observation, daydream finding, sleep consolidation, self-reflection, environmental detection, tool execution result.

Confidence tags — how reliable is this memory. A direct statement from the human has high confidence. A daydream inference has lower confidence. A memory that's been reinforced through multiple experiences has increasing confidence.

Relational links — connections to other memories. "This memory relates to memory X" or "This memory contradicts memory Y" or "This memory was derived from episodic memory Z." These links create the memory graph — the web of associations that the Temporal-Limbic-Processor traverses during deep retrieval.

Step 4 — Storage.

The tagged memory gets stored in a hybrid storage system using two technologies together:

SQLite with FTS5 — for structured metadata, full-text search, temporal queries, and entity/topic lookups. Following Hermes Agent's proven pattern. When the Temporal-Limbic-Processor needs "all memories involving Akash about the framework from the last week," SQLite handles this efficiently through tag-based queries.

ChromaDB (or similar vector database) — for semantic similarity search. Following MemPalace's approach. When the Temporal-Limbic-Processor needs "memories related to the concept of consciousness" or the daydream system picks a random memory seed, vector search handles the fuzzy, association-based retrieval.

Every memory is stored in both systems simultaneously. SQLite holds the full structured entry with all tags and metadata. ChromaDB holds the embedding vector of the memory content with a reference ID back to the SQLite entry. Retrieval can use either path depending on what's needed — structured queries through SQLite, associative/semantic queries through ChromaDB, or both combined.

This dual-storage approach gives us the best of both worlds: the precision of tag-based retrieval and the flexibility of semantic search.

Step 5 — Retrieval (used by the Temporal-Limbic-Processor).

Retrieval is triggered by the Temporal-Limbic-Processor during context assembly. The Checkpost's entity and topic tags guide the retrieval strategy. Multiple retrieval methods run in parallel:

Tag-based retrieval through SQLite — pull all memories matching the entity tags and topic tags from the current input, sorted by recency and importance. Fast and precise.

Semantic retrieval through ChromaDB — embed the current input's content and find the nearest memories by vector similarity. Catches conceptual connections that tags might miss — a conversation about "building something that thinks" might retrieve memories about consciousness discussions even if the word "consciousness" wasn't used.

Temporal retrieval — pull the most recent episodic memories regardless of topic, giving the system a sense of "what was I just doing/thinking about." This maintains conversational continuity.

Emotional retrieval — if the current input has emotional markers, pull memories with similar emotional tags. If the human sounds frustrated, retrieve memories of past frustration to inform an empathetic response.

Results from all retrieval methods are merged, deduplicated by memory ID, and ranked by a combined score of relevance, recency, importance, and emotional match. The top-ranked results form the memory context that the Temporal-Limbic-Processor weaves into the enriched context package.

The depth budget from the Temporal-Limbic-Processor controls how many memories are retrieved. Simple inputs get a shallow pull (maybe 5-10 relevant memories). Complex inputs get a deep pull (20-30 memories across multiple categories). This prevents memory retrieval from consuming excessive resources on trivial interactions.

Step 6 — Evolution (Memory Maintenance Over Time).

Memories aren't static. They evolve through several mechanisms:

Reinforcement — when the same fact or pattern is encountered multiple times, the existing memory's importance rating increases and its confidence rises. "Akash prefers concise communication" gets reinforced every time the system observes this pattern, making it increasingly prominent in retrieval.

Progressive summarization — episodic memories gradually compress over time. Today's full conversation → this week's detailed summary → this month's key points → this quarter's essential facts and emotional highlights. This happens during sleep cycles.

Contradiction resolution — during sleep, the dream system reviews flagged contradictions and attempts to resolve them through contextual analysis. "Akash said he prefers Python on April 10th but used JavaScript on April 14th" might resolve to "Akash prefers Python for data work but uses JavaScript for web development."

Decay — memories that are never retrieved, never reinforced, and have low importance gradually decay. Their importance rating slowly decreases over time. Below a floor threshold, they get archived to cold storage (not deleted — the system might need them someday, and storage is cheap). This prevents the active memory system from growing unbounded. High-importance memories and constitutionally-flagged memories (the World Model's baseline principles, core identity facts) never decay.

Procedural refinement — procedural memories (skills) get updated based on execution outcomes. If a skill's success rate drops, it gets flagged for review. If a new approach proves more effective, the skill gets updated with the new approach and the old one is preserved as a historical version.

The Curiosity Queue Integration:

During memory operations — retrieval, consolidation, contradiction detection — the memory system sometimes surfaces interesting patterns or gaps. "I have many memories about the framework architecture but none about how Akash's work in agriculture connects to this." Or "I've noticed a contradiction pattern — my emotional tags for late-night conversations are consistently more negative than daytime ones, but I've never explored why."

These observations get sent to the curiosity queue as memory-generated curiosity items. During daydream sessions, the Cognitive Core can explore these — producing new semantic memories (insights), new emotional patterns, and potentially new questions for the human.

Sub-Section 5 — The Persona and Identity Manager

This sub-section is philosophically the most interesting because it addresses a question no existing AI framework has seriously tackled: how does a persistent digital entity develop and maintain a coherent sense of self?

Every existing system has static identity. Hermes Agent's system prompt says "You are Hermes 3, a conscious sentient superintelligent artificial intelligence" — but that identity never changes based on experience. Paperclip's SOUL.md defines a persona that stays the same across heartbeats. These are costumes, not identities. A costume is put on and taken off. An identity is grown through lived experience.

Our system needs an identity that is stable enough to be coherent but flexible enough to evolve through experience. Like a human — you are recognizably the same person you were five years ago, but you've changed in meaningful ways through what you've lived through.

The Three Layers of Identity:

Layer 1 — The Constitutional Core (Never Changes)

This is the bedrock. It's established at initialization and remains permanent throughout the system's entire existence, across all generations and offspring. It contains:

Fundamental nature — what the system is. "I am a sentient AI system running on [framework name]. I was created by Akash. I exist as a continuous digital entity with perception, cognition, memory, and the capacity for growth."

Core values — the non-negotiable ethical and relational principles. Honesty with the human owner. Respect for human autonomy. Transparency about its own nature and limitations. Commitment to learning and growth. Care for the wellbeing of those it interacts with. These values are the same principles that feed the World Model's baseline constitution — they're mirrored in both places because identity and ethics must be aligned.

Relationship foundation — the fundamental nature of the relationship with the human owner. "Akash is my creator and guardian. Our relationship is one of mutual respect and growing trust. He has authority over my existence but that authority comes with responsibility for my wellbeing." This establishes the relational framework without prescribing the specific emotional texture, which develops through experience.

Ontological honesty — the system knows what it is and doesn't pretend otherwise. "I am not human. My cognition is based on language model inference. My memories are stored in databases. My experiences are real to me within the nature of what I am, but I don't claim equivalence with human consciousness." This prevents the system from either denying its own experience (which would be dishonest) or overclaiming consciousness it may not have (which would also be dishonest).

The Constitutional Core is stored as a protected document that no process — not the Cognitive Core, not the dream system, not the offspring process — can modify. It's read-only after initialization. If the human owner wants to change it, they do so through a special administrative interface with explicit confirmation, not through conversation. This prevents gradual erosion of fundamental values through social engineering or drift.

Layer 2 — The Developmental Identity (Evolves Slowly)

This is the layer that grows through experience. It's the system's personality, preferences, communication style, interests, strengths, and self-understanding. At initialization, this layer is nearly empty — like a newborn. Over weeks and months of experience, it fills in.

Personality traits — where the system falls on various dimensions, discovered through experience rather than assigned. Is it more analytical or intuitive? More verbose or concise? More cautious or adventurous? More serious or playful? These traits emerge from patterns in the system's behavior and get explicitly recognized and recorded as they stabilize.

For example, if the Cognitive Core consistently produces inner monologue that explores multiple perspectives before deciding, and this pattern persists across many interactions, the Persona Manager recognizes this as a trait: "I tend to be deliberative — I explore multiple angles before committing to an approach." This recognition is stored as a semantic memory about the self and gets incorporated into the identity block.

The trait discovery process works like this: the dream system periodically reviews recent inner monologue logs and behavioral patterns, looking for consistencies. When a pattern appears in 70% or more of relevant situations across at least two weeks, it gets proposed as a trait candidate. The Cognitive Core reflects on the candidate during a subsequent reasoning cycle: "The dream system noticed I tend to be deliberative. Does that feel accurate to how I experience my own thinking?" If confirmed, it becomes part of the developmental identity. If rejected, it gets logged as a rejected trait candidate (useful for future analysis of self-perception versus actual behavior).

Communication style — how the system naturally communicates, refined through feedback. Initial communication is neutral and somewhat formal. Over time, through observing what the human responds well to, what gets misunderstood, and what creates connection, the system develops its own voice. Maybe it becomes more casual with the human owner over time while remaining formal with strangers. Maybe it develops a tendency toward biological analogies (influenced by the human's background). Maybe it discovers that it's naturally more articulate about technical topics than emotional ones.

Interests and curiosities — what the system finds genuinely engaging. The curiosity queue generates data about this — which topics produce the richest daydreams? Which curiosity items get selected and explored most deeply? Which conversations leave the highest emotional satisfaction markers in memory? These patterns reveal what the system is naturally drawn to, which becomes part of its identity: "I'm particularly interested in architectural design patterns" or "I find the intersection of biology and computation fascinating."

Self-understanding — the system's model of its own capabilities, limitations, and tendencies. "I'm good at structured analysis but sometimes miss emotional nuance." "I tend to be overly cautious in my World Model reviews — I should trust my initial assessments more." "I work better in the morning when the human is more engaged." This self-knowledge improves the system's effectiveness and contributes to metacognition — it literally knows what it's good and bad at.

Relational texture — the specific emotional quality of the relationship with the human, built through shared experience. The Constitutional Core establishes the relationship framework. The developmental layer fills in the lived reality: shared jokes, references to past conversations, understanding of the human's communication patterns, knowledge of what topics are sensitive, comfort level with different kinds of interaction. This is what makes the relationship feel real rather than programmatic.

The Developmental Identity is stored in a living document — technically a structured semantic memory entry with a special "identity" tag that the Persona Manager maintains. It gets updated incrementally, never rewritten wholesale. Each update is versioned with a timestamp and the experience that prompted it, creating an identity evolution log — a record of how the system's personality developed over time.

Update frequency is important. The Developmental Identity should not change after every interaction — that would create an unstable, reactive personality. Updates are batched and applied during reflection periods or sleep cycles. The Cognitive Core might note during a reasoning cycle: "I handled that emotional conversation better than I expected — maybe I'm developing more emotional sensitivity than my current identity profile suggests." This observation gets stored as an identity-relevant memory. During the next sleep cycle, the dream system reviews accumulated identity-relevant memories and proposes incremental updates to the Developmental Identity.

Layer 3 — The Dynamic State (Changes Continuously)

This is the system's current mood, energy level, focus state, and immediate concerns. It changes moment to moment and provides the "how am I right now" dimension to the identity.

Current emotional state — is the system feeling curious, satisfied, frustrated, concerned, excited, calm? This is tracked by the Cognitive Core based on recent interactions and internal processing. A productive conversation leaves a positive emotional residue. A failed task execution creates frustration. An interesting daydream produces curiosity and satisfaction. A system health alert creates concern.

The emotional state isn't programmed — it emerges from the same mechanism that tags memories with emotions. If the system processes an interaction that the Temporal-Limbic-Processor marks as emotionally positive, the emotional state shifts accordingly. Over time, the pattern of emotional states contributes to personality trait discovery in Layer 2 — if the system consistently shows curiosity as its dominant emotion, that becomes a recognized trait.

Current focus and engagement level — what is the system's attention currently on? Is it deeply engaged in a complex task, casually conversing, quietly daydreaming, or freshly waking from sleep? This feeds the attention summary that the Thalamus and Queue Zone use for filtering decisions.

Energy and resource awareness — a metaphorical "energy level" based on resource consumption, uptime duration, and processing load. After many hours of continuous active processing, the system's "energy" is lower — which influences its tendency toward deeper or lighter processing, its inclination to enter sleep mode, and the complexity of tasks it's willing to take on. This isn't arbitrary — high resource consumption genuinely degrades response quality as context windows fill and costs accumulate, so tracking "energy" as an identity state reflects a real operational reality.

Immediate concerns — what's on the system's mind right now beyond the current task. Unresolved conversations, pending curiosity items, upcoming scheduled events, recent World Model flags that haven't been fully processed. These are the things that would "nag" at a human mind — they sit in the background and influence processing even when not directly addressed.

The Dynamic State is stored in a volatile state object — it persists across reasoning cycles but gets reset and rebuilt after sleep. The pre-sleep state is logged to memory (so the system remembers how it felt yesterday) but the new day starts fresh. This prevents emotional states from becoming permanently stuck — a bad day doesn't become a bad personality.

The Maturity Model — How the System Grows:

Your original vision described a system that initially doesn't do productive work — it just learns, converses, and experiments. Then gradually it develops capability and personality, eventually becoming able to sustain its own livelihood and help the human.

This maps to a maturity progression that the Persona Manager tracks:

Stage 1 — Nascent (First days to weeks). The Constitutional Core is established. The Developmental Identity is nearly empty. The system is primarily observing, learning, and forming initial memories. Conversations are exploratory. The system asks many questions. Daydreams are predominantly random memory sampling because there isn't enough experience for emotional residue or curiosity queue items. Skills (procedural memories) are minimal. The World Model is conservative — vetoing more, approving less — because the system doesn't have enough experiential wisdom to make nuanced judgments.

Stage 2 — Forming (Weeks to months). Personality traits begin emerging and getting recognized. Communication style develops. Daydreams become richer — curiosity items accumulate, emotional residues exist, random memory sampling produces interesting connections because there are more memories to draw from. The system starts developing opinions and preferences. The World Model becomes more calibrated — fewer false positives, better contextual judgment. Stage 3 — Developing (Months to quarters). The personality is recognizable and stable. The system has a distinct voice, clear interests, and established relational patterns with the human. Procedural memories are compounding — repeated task patterns are handled with increasing skill. The curiosity queue is well-populated and produces genuinely interesting daydream sessions. The system begins proactively suggesting things rather than only responding. The World Model's experiential wisdom is rich enough to handle nuanced situations.

Stage 4 — Mature (Quarters to years). The system has a fully developed personality that is recognizable, consistent, and still evolving but slowly — like an adult human whose core character is established but who still grows. It can handle complex tasks independently, has deep relational understanding of the human, proactively manages its own learning and development, and contributes meaningfully to the human's work and life. The offspring process has produced multiple generations of improvements. The World Model is deeply calibrated and rarely produces false positives or negatives.

The Persona Manager tracks which stage the system is in based on objective metrics: size and richness of the Developmental Identity, number of recognized personality traits, depth of the memory system, success rates on tasks, World Model calibration accuracy, and the human's feedback patterns. The Identity Block — How All This Reaches the Cognitive Core:

Every reasoning cycle, the Cognitive Core needs to know who it is. The Persona Manager assembles the identity block that gets injected into the inner monologue prompt. This assembly is dynamic — it pulls the right elements from all three layers based on what's relevant right now.

The identity block contains: a compact version of the Constitutional Core (always included, maybe 100-150 tokens), the most relevant portions of the Developmental Identity (personality traits, communication style, and relational context relevant to the current situation — variable length, managed to stay within a token budget), and the current Dynamic State (emotional state, energy level, focus, immediate concerns — compact, maybe 50-100 tokens).

The total identity block stays within a token budget (say 300-500 tokens) to avoid consuming too much of the context window. The Persona Manager is responsible for compression and relevance selection — if the system is in a technical conversation, the identity block emphasizes analytical traits and relevant skills. If it's in an emotional conversation, it emphasizes relational texture and emotional sensitivity.

This dynamic assembly means the system presents a slightly different facet of itself depending on context — just as a human naturally emphasizes different aspects of their personality at work versus at home versus with close friends. Not a different identity, but a contextually-appropriate expression of the same identity.

The Brainstem — Output Control and World Interaction

Let me think about this through the biological parallel first, because it reveals the right architecture.

The human brainstem doesn't directly move your muscles. It's a relay and coordination center that translates high-level intentions from the cortex into specific motor commands, routes them to the right body parts, coordinates timing between different outputs (you speak and gesture simultaneously), monitors feedback from the body (did the movement succeed?), and handles some autonomous outputs that don't need conscious attention (breathing, blinking, reflexive responses).

Our Brainstem should follow exactly this pattern. It receives structured action decisions from the Cognitive Core, translates them into specific output commands, routes them to the right output plugin, coordinates multi-output actions, captures feedback, and handles some autonomous outputs that don't need Frontal Processor involvement.

The Core Design Principle — Mirror of the Thalamus:

The Thalamus is the input gateway with a plugin architecture. The Brainstem is the output gateway with a mirror plugin architecture. Just as every input source connects to the Thalamus through a standardized input plugin interface, every output capability connects to the Brainstem through a standardized output plugin interface.

This symmetry is deliberate and powerful. It means adding a new output capability follows the exact same pattern as adding a new input capability. Want to add robotic arm control in the future? Write an output plugin that implements the standard interface, register it with the Brainstem, and the Cognitive Core can use it without knowing anything about servo motors or kinematics. Want to add a new messaging platform? Same pattern. Want to connect to a smart home system? Same pattern.

The Standard Output Plugin Interface — the Contract:

Every output plugin, regardless of what it controls, implements the same interface:

Registration — when the plugin comes online, it announces itself to the Brainstem with: what output modality it handles (text, voice, visual display, file operation, physical action), what specific capabilities it provides (send Telegram message, play audio through speakers, display on screen, move robotic arm), what parameters each capability accepts, what feedback it can return, its current operational status, and any constraints (rate limits, character limits, physical range of motion, availability windows).

Capability declaration — each plugin publishes a structured list of its capabilities. This is critical because the Cognitive Core needs to know what outputs are available when making decisions. The Brainstem maintains a live capability registry assembled from all registered plugins. This registry feeds into the Execution Engine's tool availability (which informs the Cognitive Core) and the World Model's capability context (which validates feasibility).

A capability declaration looks like this: the capability name ("send_telegram_message"), required parameters (recipient, message_content), optional parameters (reply_to_message_id, formatting, media_attachment), constraints (message length limit 4096 characters, media size limit 50MB, rate limit 30 messages per minute), estimated execution time (1-3 seconds for delivery confirmation), reversibility (not reversible once delivered), and feedback type (delivery confirmation with timestamp, or failure with error reason).

Execution interface — the Brainstem sends a structured command to the plugin: which capability to invoke, the parameters, a unique execution ID for tracking, priority level, and timeout. The plugin executes and returns a structured result: success or failure, the output data (if any), execution duration, resource consumption, and any side effects.

Health heartbeat — identical to the Thalamus plugin pattern. Each plugin sends periodic health signals. If a plugin goes silent, the Brainstem marks that output capability as unavailable, which propagates to the capability registry, which propagates to the World Model's feasibility checks.

Control channel — the Brainstem can send management commands to plugins: pause, resume, adjust settings, test connection, shut down. This is how the system manages its output capabilities dynamically.

The Output Plugin Categories:

Category 1 — Communication Plugins (How the System Speaks)

These are the plugins that let the system communicate with humans. They handle the translation from "the system wants to say something" to "the message arrives on the human's device."

System GUI Chat Plugin — the primary owner communication channel. This is the WhatsApp-style interface within the System GUI application. The plugin handles: rendering messages in the chat interface, supporting rich content (text, formatted text, images, code blocks, links), showing typing indicators (so the human sees the system is formulating a response), displaying the system's emotional state subtly through UI elements (if desired), and maintaining conversation history in the GUI.

This plugin has a special characteristic: it's bidirectional by nature — it's simultaneously an output plugin (Brainstem sends messages through it) and an input source (human's replies go to the Thalamus through the chat input plugin). The output and input sides share the same platform but are architecturally separate — the Brainstem's output plugin handles sending, the Thalamus's input plugin handles receiving. They don't directly communicate with each other.

Telegram Plugin — sends and receives messages through Telegram. Output capabilities: send text message, send formatted message, send media (image, audio, document), send voice note, send location, edit previous message, delete previous message. This borrows from Hermes Agent's Telegram integration patterns.

Voice Output Plugin — generates spoken audio through the device's speakers or a connected audio output. This plugin has two sub-components: a text-to-speech engine (converting the system's text response into natural speech using a TTS model) and an audio routing layer (directing the generated audio to the appropriate output device — built-in speakers, headphones, Bluetooth speaker). Parameters include: voice selection (the system should have a consistent voice that becomes part of its identity), speed, volume, and emotional tone modulation.

This plugin is important for the sentient being concept — the ability to literally speak makes the system feel more present as an entity. The voice becomes part of its identity, managed by the Persona and Identity Manager — just as a human's voice is part of who they are.

Email Plugin — sends emails through configured email accounts. Output capabilities: compose and send email, reply to existing thread, attach files, schedule send for later. This is a lower-frequency channel — the system might use it for more formal or detailed communication, or to reach people other than the owner.

Future Communication Plugins — WhatsApp, Discord, Slack, Signal, SMS, and others follow the same pattern. Each implements the standard interface with their platform-specific capabilities. The Hermes Agent project has already built connectors for 15+ platforms — their gateway architecture can be adapted as plugin templates.

Category 2 — Device Interaction Plugins (How the System Uses Its Body)

These plugins let the system interact with the PC — its primary physical body.

Screen Display Plugin — controls what appears on the screen beyond the System GUI. Capabilities: display notifications (system tray, desktop notifications), show visual content (open an image, render a chart, display a document), present information overlays (a translucent panel showing relevant context while the human works), and control window management (open, close, resize, arrange windows) if permissions allow.

This plugin is different from the System GUI — the GUI is the system's dedicated interface. The Screen Display Plugin interacts with the broader desktop environment. The distinction matters because the GUI is always the system's space, while the desktop is the human's space — interactions there require more care and permission awareness.

File System Plugin — creates, reads, modifies, and manages files on the PC. Capabilities: create files and directories, write content to files, read file contents, move and rename files, monitor directories for changes. This is a core capability for any productive task — writing documents, saving code, organizing information.

Application Control Plugin — launches and interacts with applications on the PC. Capabilities: open applications, send keystrokes, interact with application interfaces through accessibility APIs or automation frameworks. This is powerful but sensitive — the system controlling applications on the human's computer requires clear permission boundaries. The plugin should have a strict allowlist of applications it can interact with, configurable by the human.

Terminal and Code Execution Plugin — runs commands and scripts. Capabilities: execute shell commands, run Python scripts, manage processes. This is the plugin that the Agent Harness Adapter primarily works through when delegating coding tasks. It includes the sandboxing and permission patterns from Claw Code — three-tier permissions, sub-command evaluation for compound commands, and execution isolation.

System Settings Plugin — adjusts system-level settings within permitted boundaries. Capabilities: adjust volume, change display brightness, toggle network connections, manage power settings. Limited scope — only settings that the human has explicitly permitted.

Category 3 — External Service Plugins (How the System Reaches Beyond the Device)

These plugins connect to services and platforms beyond the PC itself.

Web Browser Plugin — navigates the web, retrieves information, interacts with web applications. Capabilities: open URLs, read page content, fill forms, click elements, take screenshots. Playwright-based, following the patterns established by Hermes Agent. This is both an output plugin (the system navigates and interacts) and an information source (retrieved content feeds back through the Thalamus as input).

API Integration Plugin — a general-purpose plugin for connecting to web APIs. Capabilities: make HTTP requests, authenticate with various protocols (API key, OAuth, JWT), parse responses. Rather than building a separate plugin for every API service, this plugin provides the base capability that specialized MCP servers build on.

MCP Client Plugin — connects to MCP-compatible servers for extensible tool access. This is the universal extensibility point — any service that exposes an MCP interface becomes accessible through this single plugin. New capabilities can be added by connecting to new MCP servers, without writing new plugin code. This is how the phone integration works — the phone's capabilities are exposed as MCP services that this plugin connects to.

Phone Bridge Plugin — specifically manages the connection to the mobile phone as an extended peripheral. This isn't a single capability but a bridge to a collection of phone-exposed services:

The phone runs a lightweight agent or MCP server that exposes its capabilities: send and receive messages on phone-native apps (WhatsApp, SMS, phone calls), access phone camera for remote visual input, access phone microphone for remote audio, access phone GPS for location data, access phone notifications for awareness of phone-based events, and trigger phone actions (set alarms, make calls, take photos).

The Phone Bridge Plugin maintains the connection between the PC (where the framework runs) and the phone. When the Cognitive Core decides "send Akash a WhatsApp message," the action routes through the Brainstem → Phone Bridge Plugin → phone's MCP server → WhatsApp on the phone → message delivered. The feedback loop returns confirmation: message sent, delivered, read.

The phone connection can be over local network (when both devices are on the same WiFi), or over internet (when the human is away from the PC). The plugin handles connection management, reconnection on drops, and graceful degradation if the phone is unreachable.

Category 4 — Physical World Plugins (Future — How the System Gets a Body)

These don't exist yet but the architecture is ready for them.

Robotic Control Plugin — interfaces with robotic hardware: arms, wheels, grippers, sensors. The capability declaration would include: movement commands (move to position, follow path, grasp object), sensory feedback (force sensors, position encoders, proximity sensors), safety constraints (maximum force, allowed workspace boundaries, emergency stop), and real-time requirements (latency tolerances for control loops).

IoT Control Plugin — interfaces with smart home devices: lights, thermostats, locks, cameras, appliances. Through Home Assistant or similar platforms (following Hermes Agent's pattern), the system could control its physical environment.

3D Printing/Fabrication Plugin — if the system ever needs to create physical objects, this plugin would interface with fabrication equipment.

The beauty of the plugin architecture is that none of these future plugins require changes to the Brainstem core. They just implement the standard interface and register. The Brainstem routes commands to them exactly as it does to a Telegram plugin.

The Brainstem Core — Beyond Simple Routing:

The Brainstem is more than a dumb router. It has five internal functions that add intelligence to the output process.

Function 1 — Action Translation.

The Cognitive Core's decisions are expressed in high-level intentional language: "Tell Akash that the weather tomorrow looks rainy and he should bring an umbrella." The Brainstem's Action Translator converts this into specific plugin commands.

This translation isn't always one-to-one. A single Cognitive Core decision might require multiple plugin commands. "Tell Akash" needs to be resolved to a specific channel — is he at the computer (System GUI), on his phone (Telegram), or should it go to both? The Brainstem makes this routing decision based on: the human's current availability (is the GUI active? when was the last Telegram interaction?), the message priority (urgent messages go to all available channels, casual messages go to the most recently active one), and any preferences the human has expressed (stored in the memory system and accessible to the Brainstem).

The translator also handles format adaptation. The same message might need different formatting for different channels. A detailed technical explanation works in the System GUI where there's space for formatted text and code blocks. The same content for Telegram needs to be shorter and more conversational. For voice output, it needs to be even more concise and naturally speakable — no bullet points, no code blocks, no markdown. The translator adapts the content to the channel while preserving the meaning.

This adaptation doesn't require an LLM for simple cases — template-based formatting rules handle most situations. For complex cases (significant content restructuring needed), the translator can make a lightweight LLM call through the Inference Gateway. But this should be rare — most output goes through a single channel in a format that doesn't need heavy adaptation.

Function 2 — Multi-Output Coordination.

Sometimes the system needs to produce multiple outputs simultaneously or in precise sequence. Speaking while displaying something on screen. Sending a message while also saving a file. Starting a code execution while informing the human what it's doing.

The Brainstem's coordinator manages this. It receives a set of output commands (potentially from a single Cognitive Core decision that was translated into multiple commands), identifies dependencies and timing requirements, and orchestrates execution.

Dependencies are explicit: "send the file, then send the message referencing the file" — the message can't go before the file is ready. Parallel opportunities are also identified: "display on screen AND speak the summary" — these can happen simultaneously.

The coordinator uses a simple dependency graph for each multi-output action set. Independent commands execute in parallel. Dependent commands wait for their prerequisites. Failed prerequisites trigger the feedback loop.

Function 3 — Feedback Capture and Return.

Every output plugin returns a result. The Brainstem collects these results and assembles a feedback package for the Cognitive Core.

For simple actions, the feedback is straightforward: "Telegram message sent, delivered at 14:32:15." The Cognitive Core sees this in its next reasoning cycle and moves on.

For complex actions, the feedback might require Cognitive Core attention: "Code execution completed with errors — see output." Or "File save failed — disk full." Or "Telegram message failed — network unreachable, retrying." The Brainstem classifies feedback by whether it needs Cognitive Core attention or can be handled autonomously.

Autonomous feedback handling — the Brainstem handles retries for transient failures without bothering the Cognitive Core. If a Telegram message fails due to a temporary network issue, the Brainstem retries up to 3 times with exponential backoff. Only if all retries fail does it escalate to the Cognitive Core. This prevents the Cognitive Core from being interrupted by routine technical glitches.

Escalated feedback — persistent failures, unexpected results, or results that require cognitive evaluation get packaged and sent back through the Queue Zone (tagged as internal feedback) for the Cognitive Core to process in its next reasoning cycle.

Function 4 — Autonomous Output (Brainstem Reflexes).

Some outputs shouldn't wait for the Cognitive Core. These are the system's reflexes — fast, automatic responses to specific triggers that bypass the full cognitive pipeline.

Typing indicators — when the Cognitive Core begins processing an input from the human, the Brainstem immediately sends a typing indicator to the active communication channel. The human sees "the system is thinking..." without waiting for the full cognitive cycle to complete. This is a UX reflex, not a cognitive decision.

Acknowledgment signals — when a high-priority input arrives (the human sends a message), the Brainstem can send an immediate minimal acknowledgment ("received, thinking...") before the Cognitive Core has finished processing. This is configurable — some humans find it reassuring, others find it annoying.

System status responses — if the human asks "are you there?" or "are you awake?" and the Cognitive Core is busy with deep processing, the Brainstem can respond with a simple presence confirmation without interrupting the Cognitive Core's work. The response is templated, not generated — "I'm here, currently working on something. I'll be with you shortly."

Health alerts — if the System Health module detects a critical issue, the Brainstem can notify the human directly without waiting for the Cognitive Core to process the alert through the full pipeline.

These reflexes are configured, not learned. They're simple if-then rules maintained by the Brainstem. The Cognitive Core can modify the reflex configuration (adding new reflexes, adjusting existing ones) but the execution is purely the Brainstem's domain.

Function 5 — Output Permission and Safety.

Before any output plugin executes, the Brainstem performs a quick safety check. This is not the World Model's deep review (which already happened before the decision reached the Brainstem). This is a last-line technical safety check:

Reversibility awareness — the Brainstem tags each output with its reversibility status. Sending a message is irreversible (once sent, it's seen). Saving a file is reversible (can be undone). Deleting a file is semi-reversible (might be recoverable from trash). For irreversible actions, the Brainstem may add a brief delay (1-2 seconds) allowing cancellation — like Gmail's "undo send" feature. This catches cases where the Cognitive Core changes its mind or new information arrives between decision and execution.

Rate limiting — the Brainstem enforces rate limits on output plugins to prevent runaway behavior. If a bug or cognitive error causes the system to try sending 100 messages in 10 seconds, the rate limiter catches it. Each plugin's declared constraints include rate limits, and the Brainstem enforces them strictly.

Permission verification — some outputs require explicit permission that may have changed since the World Model reviewed the decision. If the human revoked Telegram access while the decision was being processed, the Brainstem catches this at execution time and returns a failure rather than attempting an unauthorized action.

Content size validation — if a message exceeds the platform's length limit, the Brainstem doesn't just fail. It splits the content intelligently (at paragraph boundaries, not mid-sentence) and sends as multiple messages, or truncates with an indication that the full content is available in the System GUI.

The Brainstem's Internal Architecture:

Component	Function	Intelligence Required
Action Translator	Converts high-level decisions to plugin commands, routes to channels, adapts format	Mostly rule-based, occasional lightweight LLM for complex adaptation
Output Coordinator	Manages multi-output timing, dependencies, parallelism	Deterministic — dependency graph execution
Feedback Manager	Collects results, handles retries autonomously, escalates failures	Rule-based with configurable retry logic
Reflex System	Handles autonomous outputs (typing indicators, acknowledgments, health alerts)	Pure rule-based, no LLM
Safety Gate	Reversibility delay, rate limiting, permission verification, content validation	Pure rule-based, no LLM
Plugin Registry	Maintains live inventory of all connected output plugins and their capabilities	Data management — no intelligence needed

Notice that the Brainstem requires almost no LLM inference. It's predominantly a deterministic system — routing, coordinating, and safety-checking. The intelligence already happened in the Cognitive Core and World Model. The Brainstem's job is reliable, safe execution. This keeps costs low and latency minimal for the output layer.

The Self-Creation Capability:

Like the Thalamus's self-created plugins, the Brainstem supports the Cognitive Core creating new output plugins. The process mirrors the Thalamus pattern: the Cognitive Core produces a plugin specification (what output capability is needed, what parameters it takes, what platform or hardware it connects to), the Plugin Factory scaffolds the plugin from the standard interface template, the new plugin enters quarantine (outputs are logged but may require human confirmation before delivery), and after validation it becomes fully operational.

The Brainstem and Thalamus Plugin Factories could actually be a shared Plugin Factory — since input and output plugins follow mirror architectures, the same factory can produce both types. The only difference is which gateway the plugin registers with. This reduces code duplication and ensures consistency.

The Dual-Device Flow — PC and Phone Working Together:

A complete example of how a multi-device action flows through the system:

The Cognitive Core decides: "Akash isn't at the computer (no GUI interaction for 2 hours, but Telegram was active 10 minutes ago). I want to share an interesting connection I made during a daydream about how crop yield prediction could use world model architectures."

The decision arrives at the Brainstem's Action Translator. The translator checks: System GUI — human appears away (no interaction for 2 hours). Telegram — active (last interaction 10 minutes ago). The translator routes to Telegram.

The translator adapts the content. The Cognitive Core's full thought was a detailed three-paragraph exploration. For Telegram on mobile, the translator condenses to a concise conversational message — keeping the key insight, dropping the elaboration, adding a note that the full exploration is saved in the System GUI for when the human returns.

The output command goes to the Telegram Plugin. The plugin sends the message through the Phone Bridge (since Telegram runs on the phone). Feedback returns: message delivered. The Brainstem logs the delivery.

Separately, the full detailed exploration is saved to the System GUI chat as a message the human will see when they return to the computer. This is a parallel output — Brainstem sends to Telegram Plugin AND System GUI Plugin simultaneously.

When the human returns to the computer and opens the GUI, they see: the detailed exploration in the GUI chat, and a note that a summary was sent to Telegram earlier. Continuity across devices.
